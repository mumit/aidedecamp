# Hosted Google OAuth transaction

This contract governs linking a Google Workspace account to an authenticated
hosted Attune principal. It is not the hosted sign-in flow. OAuth remains
disabled until every boundary below has live evidence.

## Trust boundaries

The authenticated control plane may create a transaction only from a verified,
recent browser session and a canonical pending connector. The public callback
scrubber has no tenant, database, secret, KMS, queue, OAuth-client, or provider
authority. A separate private exchange service resolves an opaque transaction;
the secret broker alone reads the OAuth client secret, exchanges the code, and
persists an encrypted credential.

The callback supplies no tenant, principal, connector, redirect URI, scope, or
provider authority. Those values come from the canonical transaction record.

## Transaction creation

For each attempt, trusted code generates independent cryptographic values:

- 256-bit `state`, storing only its SHA-256 hash;
- a 256-bit callback binding, storing only its SHA-256 hash;
- an RFC 7636 verifier with an S256 challenge;
- a 256-bit OIDC nonce, storing only its SHA-256 hash; and
- an opaque transaction UUID generated by PostgreSQL.

The tenant-scoped record binds tenant, authenticated principal, pending Google
connector, canonical requested `google.oauth.install` credential intent, exact
redirect URI, exact ordered scopes, state hash, binding hash, nonce hash, PKCE
verifier, creation time, ten-minute expiry, attempt count, and state. The
verifier is short-lived and protected by the private CMEK-encrypted database
and RLS; it never enters URLs, cookies, logs, analytics, or client responses
after the authorization redirect is created.

The browser receives a `Secure`, `HttpOnly`, `SameSite=Lax` callback-only
binding cookie. It is separate from the main login session and carries no
tenant or principal data. The authorization request uses an exact HTTPS
redirect, response type `code`, PKCE S256, transaction-specific state and
nonce, offline access, explicit consent where required for a refresh token, and
only reviewed scopes.

## Callback and consumption

1. The dedicated public scrubber bounds the query length and accepts only the
   exact host, method, and callback path. Its request-log planes are excluded by
   dedicated resource identity as specified in `security-architecture.md`.
2. The scrubber requires exactly one bounded `code`, `state`, and binding
   cookie. Provider errors are normalized without reflecting their fields. It
   clears the binding cookie and sends the three opaque values over an
   authenticated private request; it never receives the main login session.
3. The private exchange service hashes state and binding and calls one narrow
   `SECURITY DEFINER` lease function. The function searches across tenants but
   returns a record only when both hashes match, the transaction is pending or
   its lease expired, the connector remains pending, and expiry has not passed.
   No runtime role gets cross-tenant table access.
4. The private service passes the canonical install intent, authorization code,
   PKCE verifier, nonce hash, redirect URI, and provider identifier to one fixed
   secret-broker operation. The broker supplies the client secret, uses the
   fixed Google token endpoint, forbids redirects and ambient proxies, validates
   the ID token issuer/audience/nonce/time claims, minimizes the account
   identity, envelope-encrypts the refresh token, and returns only an opaque
   connector result.
5. The transaction is finalized once. A crash or ambiguous provider/vault
   result fails closed and requires a fresh user ceremony; an authorization
   code is never durably stored for automatic retry.

Success and refusal produce canonical content-free audit events. Codes, tokens,
verifiers, state, nonce, binding cookies, raw provider errors, email addresses,
and callback URLs never enter audit metadata.

## Required negative evidence

- guessed, malformed, missing, duplicate, expired, and replayed state;
- missing, wrong, duplicate, or replayed binding cookie;
- cross-tenant state/binding/connector substitution;
- connector no longer pending or principal/session changed;
- redirect, scope, provider, issuer, audience, nonce, and PKCE mismatch;
- duplicate callback and concurrent lease attempts;
- callback query/referrer/log leakage and remote log-query self-retention;
- broker redirect, proxy, DNS, SSRF, oversized response, and token reflection;
- crash before exchange, after exchange, and during vault/audit finalization; and
- revocation or tenant suspension during the transaction.

The OAuth client is configured only after global route convergence and
synthetic non-retention evidence. Creating a client, receiving a 303, or
successfully exchanging a code is not by itself authorization to admit customer
data.
